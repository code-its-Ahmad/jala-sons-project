SECTION 1: EXECUTIVE SUMMARY
================================================================================

Jalal Sons Smart Operations Platform is a full-stack, AI-powered retail and food
delivery system designed to eliminate the operational friction experienced by
traditional grocery and restaurant chains. The platform integrates predictive
machine learning, autonomous multi-agent AI, real-time geolocation tracking,
and vector-based semantic search into a unified, cost-effective solution built
entirely on free-tier and open-source technologies.

The system serves three primary user roles:

  CUSTOMER (End-User)
    Places orders, tracks delivery live on a map, receives AI-personalized
    substitution offers when items run out, and gets real-time status updates.

  RIDER (Delivery Agent)
    Receives auto-assigned orders from the AI Dispatcher, shares live GPS
    coordinates, follows turn-by-turn navigation, and updates delivery status.

  ADMIN (Operations Manager)
    Monitors live order pipelines, manages AI-predicted inventory alerts,
    oversees the LangChain Dispatcher Agent, and configures AI parameters.

Core Technology Pillars:
  - FastAPI (async Python backend)
  - LangChain (multi-agent AI orchestration)
  - Gemini 2.5 Flash (LLM + embeddings)
  - TensorFlow / Keras (demand forecasting LSTM)
  - Supabase (ALL data: PostgreSQL + Auth + Realtime + pgvector + Storage)
  - Apify (data ingestion from restaurant & grocery sources)
  - Leaflet + OSM + Nominatim + OpenRouteService + GraphHopper (all-free maps)
  - Next.js 14 + Tailwind CSS (frontend)

The entire data layer is consolidated into a single Supabase project. No
separate Redis, Pinecone, Firebase, or Auth0 is required.

SECTION 2: PROBLEM STATEMENT (DETAILED)
================================================================================

2.1  INVENTORY DISCONNECTION
     Traditional POS systems do not synchronize with digital storefronts in
     real-time. A product shown as "Available" online may already be sold out
     in the physical store. Consequences:
       - Order cancellations at the final checkout stage
       - Manual staff intervention required
       - Customer trust erosion and churn
       - Direct revenue leakage per cancelled order

2.2  NO INTELLIGENT SUBSTITUTION LOGIC
     When a checkout item is out of stock, current systems cancel or require a
     phone call to support. There is no automated, context-aware fallback that:
       - Understands the nutritional or culinary context of the missing item
       - Suggests semantically relevant alternatives
       - Completes the sale with minimal disruption and customer effort

2.3  MANUAL RIDER DISPATCH
     Assigning delivery riders manually is slow, inconsistent, and unscalable.
     During peak hours, dispatchers become the primary bottleneck. Factors like
     proximity, current order load, and vehicle type are inconsistently applied,
     resulting in:
       - Delayed order pickups
       - Uneven rider workload distribution
       - Longer delivery times and frustrated customers

2.4  STATIC ROUTE PLANNING
     Fixed delivery routes ignore real-time conditions. The result is:
       - Unnecessarily long delivery times
       - Higher fuel costs per delivery
       - No ETA feedback to the customer until the door

2.5  NO PREDICTIVE CAPABILITY
     Operations react to stockouts rather than preventing them. Without demand
     forecasting, procurement decisions are reactive and often inaccurate, leading
     to either waste (overstock) or missed sales (stockout).

2.6  POOR CUSTOMER EXPERIENCE ON TRACKING
     Current implementations offer no live GPS tracking. Customers receive only
     a static "Your order is on its way" message, leading to inbound support
     calls and reduced satisfaction scores.

SECTION 3: COMPLETE TECH STACK & RATIONALE
================================================================================

3.1  BACKEND FRAMEWORK
     Language    : Python 3.11+
     Framework   : FastAPI 0.110+
     Rationale   : Async-first architecture, OpenAPI auto-documentation,
                   native WebSocket support, high concurrency under uvicorn,
                   seamless integration with all Python ML/AI libraries.

3.2  AI AGENT ORCHESTRATION
     Framework   : LangChain 0.2+
     Rationale   : Enables multi-agent tool orchestration, chain-of-thought
                   reasoning, structured agent memory, and tool-use pipelines.
                   Fully open-source, zero licensing cost.

3.3  CORE LLM / GENERATIVE AI
     Provider    : Google Gemini API
     Model       : gemini-2.5-flash (completions)
                   text-embedding-004 (768-dimension embeddings)
     Rationale   : Free tier with 1M tokens/day quota. Fastest Gemini model.
                   Superior at semantic embedding, context-aware recommendation,
                   and natural language generation for user-facing messages.

3.4  MACHINE LEARNING
     Library     : TensorFlow 2.16+ with Keras
     Model Type  : LSTM (Long Short-Term Memory) Neural Network
     Rationale   : Open-source. Purpose-built for time-series demand forecasting.
                   LSTM excels at capturing seasonal and weekly sales patterns.

3.5  PRIMARY DATABASE, AUTH, REALTIME, VECTOR, STORAGE
     Platform    : Supabase (Free Tier: 500MB DB, 50k MAU, 5GB bandwidth)
     Services    :
       PostgreSQL  : All relational data (17 tables defined in Section 5)
       Auth        : JWT-based authentication (email/password + Google OAuth)
       Realtime    : WebSocket change data capture (CDC) for live UI updates
       pgvector    : 768-dim embedding vectors for semantic product search
       Storage     : Product images, rider avatars, ML model artifacts
     Rationale   : Single platform replaces Redis, Pinecone, Firebase, Auth0.
                   All data in one place; RLS-enforced security; zero extra cost.

3.6  DATA INGESTION
     Platform    : Apify (Free Tier: $5/month credit)
     Scrapers    :
       Restaurant : apify.com/its_raghav/restaurant-menu-scraper
       Grocery    : apify.com/piotrv1001/instacart-scraper
     Rationale   : Structured menu/product data without custom scraper
                   maintenance. Outputs clean JSON via webhook to FastAPI.

3.7  MAPPING & GEOLOCATION (ALL FREE)
     Map Rendering      : Leaflet.js + OpenStreetMap tile server
     User Location      : Browser Geolocation API (navigator.geolocation)
     Place Search       : Nominatim (OpenStreetMap) — no key required
     Reverse Geocoding  : Nominatim
     Route Calculation  : OpenRouteService (Free: 2,000 req/day)
     Alternative Routes : GraphHopper (Free: 500 req/day)
     Distance Matrix    : OpenRouteService
     Turn-by-Turn Nav   : GraphHopper Directions API

3.8  FRONTEND
     Framework          : Next.js 14 (React, App Router)
     Styling            : Tailwind CSS + ShadCN UI component library
     State Management   : Zustand (lightweight, no boilerplate)
     Realtime Client    : Supabase JS Client (@supabase/supabase-js)
     Map Client         : Leaflet.js via react-leaflet wrapper
     Charts & Analytics : Recharts (admin dashboard — free, React-native)
     Animations         : Framer Motion (page transitions, modals, map markers)

3.9  BACKGROUND TASKS & SCHEDULING
     Async Workers      : FastAPI BackgroundTasks (inline with request lifecycle)
     Scheduled Jobs     : APScheduler (Python) for:
                            - ML model retraining (weekly, Sunday 02:00 AM)
                            - Inventory prediction scan (every 30 minutes)
                            - Rider location cleanup (daily, archive >30 days)

3.10 EMAIL NOTIFICATIONS
     Service            : Resend API (Free: 3,000 emails/month)
     Templates          : HTML email for order confirmation, delivery receipt,
                          password reset

3.11 DEPLOYMENT (ALL FREE TIER)
     Backend            : Railway.app or Render.com (free tier Docker container)
     Frontend           : Vercel (free tier, optimized for Next.js)
     ML Models          : Stored in Supabase Storage, loaded at server startup

SECTION 5: SUPABASE DATABASE SCHEMA (COMPLETE — ALL 17 TABLES)
================================================================================

All tables reside in Supabase PostgreSQL with Row-Level Security (RLS) enabled.
The service_role key (used by FastAPI backend) bypasses RLS.
The anon/authenticated key (used by frontend) is governed by RLS policies.

────────────────────────────────────────────────────────────────────────────────
TABLE 1: users
--------------------------------------------------------------------------------
  id               UUID        PRIMARY KEY DEFAULT gen_random_uuid()
  email            TEXT        UNIQUE NOT NULL
  full_name        TEXT
  phone            TEXT
  role             TEXT        CHECK (role IN ('customer','rider','admin'))
                               DEFAULT 'customer'
  avatar_url       TEXT
  default_address  TEXT
  default_lat      FLOAT8
  default_lng      FLOAT8
  created_at       TIMESTAMPTZ DEFAULT now()
  updated_at       TIMESTAMPTZ DEFAULT now()

  RLS POLICY: Users can SELECT and UPDATE only their own row (auth.uid() = id).
              Admins can SELECT all rows.

────────────────────────────────────────────────────────────────────────────────
TABLE 2: stores
--------------------------------------------------------------------------------
  id               UUID        PRIMARY KEY DEFAULT gen_random_uuid()
  name             TEXT        NOT NULL   -- e.g. "Jalal Sons - Gulberg"
  type             TEXT        CHECK (type IN ('grocery','restaurant','hybrid'))
  address          TEXT
  latitude         FLOAT8
  longitude        FLOAT8
  phone            TEXT
  open_time        TIME
  close_time       TIME
  is_active        BOOLEAN     DEFAULT true
  created_at       TIMESTAMPTZ DEFAULT now()

  RLS POLICY: All authenticated users can SELECT active stores.
              Only admin role can INSERT/UPDATE/DELETE.

────────────────────────────────────────────────────────────────────────────────
TABLE 3: categories
--------------------------------------------------------------------------------
  id               UUID        PRIMARY KEY DEFAULT gen_random_uuid()
  store_id         UUID        REFERENCES stores(id) ON DELETE CASCADE
  name             TEXT        NOT NULL   -- e.g. "Vegetables", "Burgers"
  slug             TEXT        UNIQUE
  parent_id        UUID        REFERENCES categories(id)  -- subcategory support
  type             TEXT        CHECK (type IN ('grocery','restaurant'))
  icon_url         TEXT
  sort_order       INTEGER     DEFAULT 0
  is_active        BOOLEAN     DEFAULT true

  RLS POLICY: All authenticated users can SELECT. Admin only for writes.

────────────────────────────────────────────────────────────────────────────────
TABLE 4: products
--------------------------------------------------------------------------------
  id               UUID        PRIMARY KEY DEFAULT gen_random_uuid()
  store_id         UUID        REFERENCES stores(id) ON DELETE CASCADE
  category_id      UUID        REFERENCES categories(id)
  name             TEXT        NOT NULL
  description      TEXT
  price            NUMERIC(10,2) NOT NULL
  unit             TEXT                   -- e.g. "kg", "piece", "litre"
  image_url        TEXT
  is_active        BOOLEAN     DEFAULT true
  tags             TEXT[]                 -- ['spicy','vegan','gluten-free']
  nutritional_info JSONB                  -- {calories, protein, carbs, fat, ...}
  embedding        VECTOR(768)            -- pgvector column (Gemini embedding)
  source_url       TEXT                   -- Apify origin URL
  apify_item_id    TEXT                   -- Apify deduplication key
  created_at       TIMESTAMPTZ DEFAULT now()
  updated_at       TIMESTAMPTZ DEFAULT now()

  INDEXES:
    - products(store_id)
    - products(category_id)
    - products(embedding) USING ivfflat (vector_cosine_ops) WITH (lists=100)
    - products(is_active, store_id)

  RLS POLICY: All authenticated users can SELECT active products.
              Admin and service_role for writes.

────────────────────────────────────────────────────────────────────────────────
TABLE 5: inventory
--------------------------------------------------------------------------------
  id                   UUID        PRIMARY KEY DEFAULT gen_random_uuid()
  product_id           UUID        REFERENCES products(id) ON DELETE CASCADE UNIQUE
  quantity             INTEGER     NOT NULL DEFAULT 0
  reserved_quantity    INTEGER     DEFAULT 0  -- soft-hold for active carts
  unit                 TEXT
  low_stock_threshold  INTEGER     DEFAULT 10
  critical_threshold   INTEGER     DEFAULT 3
  last_restocked_at    TIMESTAMPTZ
  updated_at           TIMESTAMPTZ DEFAULT now()

  DERIVED FIELD (computed in queries):
    available_quantity = quantity - reserved_quantity

  RLS POLICY: Customers cannot directly SELECT inventory (accessed via API only).
              Riders and admins can SELECT. Service_role handles all writes.

────────────────────────────────────────────────────────────────────────────────
TABLE 6: inventory_predictions
--------------------------------------------------------------------------------
  id                   UUID        PRIMARY KEY DEFAULT gen_random_uuid()
  product_id           UUID        REFERENCES products(id) ON DELETE CASCADE
  predicted_demand_24h FLOAT8
  predicted_demand_48h FLOAT8
  predicted_demand_7d  FLOAT8
  confidence_score     FLOAT8      -- 0.0 to 1.0
  predicted_stockout_at TIMESTAMPTZ -- when current stock will run out at predicted rate
  model_version        TEXT
  generated_at         TIMESTAMPTZ DEFAULT now()

  INDEXES: inventory_predictions(product_id), inventory_predictions(predicted_stockout_at)
  RLS POLICY: Admin role only for SELECT. Service_role for writes.

────────────────────────────────────────────────────────────────────────────────
TABLE 7: orders
--------------------------------------------------------------------------------
  id               UUID        PRIMARY KEY DEFAULT gen_random_uuid()
  user_id          UUID        REFERENCES users(id)
  store_id         UUID        REFERENCES stores(id)
  rider_id         UUID        REFERENCES riders(id)   -- nullable until assigned
  status           TEXT        CHECK (status IN (
                                 'PENDING', 'CONFIRMED', 'PREPARING',
                                 'READY_FOR_PICKUP', 'RIDER_ASSIGNED',
                                 'OUT_FOR_DELIVERY', 'DELIVERED', 'CANCELLED'
                               )) DEFAULT 'PENDING'
  total_amount     NUMERIC(10,2)
  delivery_address TEXT
  delivery_lat     FLOAT8
  delivery_lng     FLOAT8
  delivery_notes   TEXT
  estimated_eta    INTEGER     -- minutes (updated by ETA engine)
  payment_method   TEXT        CHECK (payment_method IN ('cod','card','wallet'))
  payment_status   TEXT        CHECK (payment_status IN ('pending','paid','refunded'))
                               DEFAULT 'pending'
  cancellation_reason TEXT
  created_at       TIMESTAMPTZ DEFAULT now()
  updated_at       TIMESTAMPTZ DEFAULT now()

  INDEXES: orders(user_id), orders(rider_id), orders(status), orders(store_id)
  REALTIME: ENABLED on this table (CDC-based broadcasts to order channels)

  RLS POLICY:
    Customers: SELECT/UPDATE own orders (where user_id = auth.uid())
    Riders: SELECT orders where rider_id = their rider profile's id
    Admins: Full access
    All status updates go through FastAPI backend (service_role)

6.1  TENSORFLOW DEMAND FORECASTING MODEL

     Architecture : LSTM Neural Network with 3 output heads
     Input Window : 90-day rolling historical sales per product

     INPUT FEATURES (per time step):
       - units_sold_that_hour        : FLOAT (target variable in training)
       - day_of_week_sin             : FLOAT (cyclical encoding sin(2π*d/7))
       - day_of_week_cos             : FLOAT (cyclical encoding cos(2π*d/7))
       - hour_of_day_sin             : FLOAT (cyclical encoding for restaurants)
       - hour_of_day_cos             : FLOAT
       - week_of_month               : INTEGER (1–5)
       - is_holiday_flag             : BOOLEAN (local Pakistani holidays)
       - weather_code                : INTEGER (0=clear, 1=rain, 2=storm)
       - cumulative_sales_today      : FLOAT (running total to current hour)

     OUTPUT HEADS:
       - predicted_demand_24h        : FLOAT
       - predicted_demand_48h        : FLOAT
       - predicted_demand_7d         : FLOAT
       (All trained simultaneously with multi-task learning)

     AUXILIARY OUTPUT:
       - confidence_score            : FLOAT (0.0–1.0), derived from prediction
                                       variance across the 3 output heads

     TRAINING SCHEDULE:
       Full Retrain   : Every Sunday 02:00 AM (APScheduler job)
       Incremental    : Every 6 hours (fine-tune on last 24h sales data only)
       Data Source    : sales_history table
       Artifact Path  : Supabase Storage "ml-models/demand_v{version}.keras"

     INFERENCE FLOW:
       1. APScheduler fires every 30 minutes
       2. FastAPI loads model from local /tmp/ (pre-loaded at startup)
       3. Fetch recent 90-day sales_history for each product
       4. Run model.predict() in batch (all products per store)
       5. Compute predicted_stockout_at = now() + (current_qty / predicted_demand_rate)
       6. Upsert into inventory_predictions table
       7. Trigger LangChain Restock Alert Agent for products where
          predicted_stockout_at < now() + 48 hours

     CART VALIDATION INFERENCE:
       Triggered on every POST /inventory/validate call (real-time, per product)
       Uses cached prediction from inventory_predictions table (< 30 min old)
       if no cached prediction exists: runs single-product inference on demand

6.2  LANGCHAIN AGENT: DISPATCHER CLERK
────────────────────────────────────────────────────────────────────────────────

     Purpose      : Autonomous, zero-human-touch rider assignment
     LLM          : Gemini 2.5 Flash
     Memory       : ConversationBufferWindowMemory (last 5 steps for context)
     Trigger      : Supabase Realtime CDC on orders.status → 'READY_FOR_PICKUP'

     TOOLS AVAILABLE TO AGENT:
       get_available_riders(store_lat, store_lng, radius_km)
         → Queries riders table: is_online=true, is_available=true, load < 3
         → Returns: [{rider_id, name, lat, lng, distance_km, rating, load}]

       calculate_distance_matrix(store_coords, rider_coords_list)
         → Calls OpenRouteService Distance Matrix API
         → Returns: {rider_id: duration_seconds} map

       score_riders(riders_list)
         → Formula: (1/distance_km * 0.5) + (avg_rating/5 * 0.3) + ((3-load)/3 * 0.2)
         → Normalizes all scores, returns ranked list

       assign_rider(rider_id, order_id)
         → Sets orders.rider_id = rider_id, orders.status = 'RIDER_ASSIGNED'
         → Sets riders.active_order_count += 1, riders.is_available = false

       notify_rider_of_assignment(rider_id, order_id)
         → Inserts into notifications table (type='rider_assigned')
         → Supabase Realtime delivers to rider frontend instantly

       start_acceptance_timer(rider_id, order_id, timeout_seconds)
         → Schedules a FastAPI background task for 60 seconds
         → If rider doesn't ACCEPT within window: calls reassign_on_timeout()

       reassign_on_timeout(order_id, excluded_rider_ids)
         → Re-runs the agent excluding previously tried riders

     STEP-BY-STEP AGENT LOGIC:
       Step 1 → Fetch available riders within 5km using get_available_riders()
       Step 2 → If no riders found at 5km: expand to fallback radius (10km)
       Step 3 → Call calculate_distance_matrix() for accurate travel distances
       Step 4 → Call score_riders() to rank candidates
       Step 5 → Call assign_rider() with top-scored rider
       Step 6 → Call notify_rider_of_assignment()
       Step 7 → Start 60-second acceptance timer
       Step 8 → If ACCEPTED: log success, done
                If REJECTED or TIMEOUT: call reassign_on_timeout(), repeat from Step 4
       Step 9 → After 3 failed attempts: set order to 'PENDING_RIDER', alert admin
       
     EDGE CASE HANDLING:
       No riders within 10km      : Alert admin, queue order with PENDING_RIDER status
       All riders at max load      : Retry every 2 minutes until a rider frees up
       Severe weather             : Set admin_settings.ai_dispatch_enabled = false flag,
                                    push emergency alert to admin UI for manual override
       Rider network disconnected : Timeout treated as rejection, auto-reassign

     LOG OUTPUT: All steps logged to agent_logs table (type='dispatcher')

6.3  LANGCHAIN AGENT: SUBSTITUTION ENGINE
────────────────────────────────────────────────────────────────────────────────

     Purpose      : Find and offer intelligent product replacements when items
                    go out of stock during checkout
     LLM          : Gemini 2.5 Flash
     Memory       : None (stateless per invocation)
     Trigger      : POST /order/checkout when ordered_quantity > inventory.quantity

     TOOLS AVAILABLE TO AGENT:
       generate_embedding(text)
         → Calls Gemini text-embedding-004 API
         → Returns 768-dimension float vector

       search_similar_products(embedding, store_id, top_k=10)
         → Executes pgvector cosine similarity query on products table
         → Filters: is_active=true, store_id match, stock > 0
         → Returns: [{product_id, name, price, similarity_score, tags, nutritional_info}]

       filter_candidates(candidates, price_ceiling_factor=1.5, exclude_tags=[])
         → Removes candidates priced > original_price * 1.5
         → Removes candidates with any excluded tags

       evaluate_substitution_quality(original_product, candidate_product)
         → LLM prompt: "Given that a customer ordered {original}, evaluate if
           {candidate} is a suitable substitute based on taste, nutrition, and
           culinary use. Rate suitability 1-10 and explain in 1 sentence."
         → Returns: {suitability_score: int, explanation: str}

       calculate_offer_price(original, candidate)
         → If candidate.price > original.price: apply discount to match original
         → If candidate.price <= original.price: maintain candidate price
         → Returns: {final_price, discount_percentage}

       push_substitution_offer(order_id, original_id, candidate_id, reasoning, discount)
         → Inserts into substitution_offers table
         → Supabase Realtime delivers to customer frontend

     STEP-BY-STEP AGENT LOGIC:
       Step 1 → Detect deficit: ordered_qty > inventory.available_quantity
       Step 2 → Build embedding text: "{name}. {description}. Tags: {tags}. {nutritional_info}"
       Step 3 → Call generate_embedding() for the out-of-stock product
       Step 4 → Call search_similar_products() → top 10 candidates
       Step 5 → Call filter_candidates() to remove expensive/incompatible options
       Step 6 → For top 5 remaining: call evaluate_substitution_quality() in parallel
       Step 7 → Sort by (suitability_score * 0.6 + similarity_score * 0.4)
       Step 8 → Select top 3 as the offer set
       Step 9 → Call calculate_offer_price() per selected candidate
       Step 10 → For each of top 3: call push_substitution_offer()
       Step 11 → All 3 offers inserted (rank 1, 2, 3) → Realtime fires to frontend
       Step 12 → Frontend renders modal with 3 options; user selects one

     OFFER EXPIRY:
       If user does not respond in 5 minutes:
       → Auto-reject all substitution_offers (user_response = 'rejected')
       → Remove item from order, recalculate order total
       → Notify user: "Your item has been removed. Total updated."

     LOG OUTPUT: Logged to agent_logs table (type='substitution')

SECTION 14: WEBSOCKET COMMUNICATION SPECIFICATION
================================================================================

FastAPI WebSocket Endpoint: ws://api/rider/location/{rider_id}

  CONNECTION LIFECYCLE:
    1. Rider app opens WebSocket connection after accepting an order
    2. FastAPI validates rider_id from JWT in connection query param
    3. Connection accepted: server registers rider_id → WebSocket mapping
    4. Rider browser starts watchPosition() → sends message every 5 seconds
    5. Message format (JSON string):
       {"lat": 31.4820, "lng": 74.3150, "speed": 28.5, "heading": 180.0, "ts": "ISO8601"}
    6. FastAPI handler:
       a. Parses message
       b. Inserts into rider_locations table
       c. Updates riders.current_latitude and current_longitude
       d. Computes ETA delta (if position changed significantly: > 50m)
       e. If ETA changed: updates orders.estimated_eta, generates Gemini message
    7. Rider app also receives incoming messages (optional):
       {"type": "assignment_update", "status": "..."}  for new order assignments
    8. Disconnection: on WebSocket close, FastAPI removes rider from active map
       Rider GPS streaming naturally stops when watchPosition() is cleared

  RECONNECTION STRATEGY:
    Client-side: Exponential backoff (1s, 2s, 4s, 8s, max 30s) on disconnect
    Server-side: Accepts new connection and resumes processing from latest GPS

15.2  ROW-LEVEL SECURITY (RLS) POLICIES

  users Table:
    SELECT: auth.uid() = id  (users see own row only)
    UPDATE: auth.uid() = id
    Admin override: role check in JWT → service_role for admin reads

  orders Table:
    SELECT for customers: auth.uid() = user_id
    SELECT for riders: EXISTS(SELECT 1 FROM riders WHERE user_id=auth.uid() AND id=orders.rider_id)
    SELECT for admins: (auth.jwt()->'user_metadata'->>'role') = 'admin'
    INSERT/UPDATE: service_role only (all order mutations go through FastAPI backend)

  inventory Table:
    SELECT: No direct customer access (served through API only)
    SELECT for admins and riders: role check
    All writes: service_role only

  riders Table:
    SELECT: All authenticated users can see basic rider info (name, vehicle)
    UPDATE: auth.uid() = user_id (rider can update own availability)
    Full access: service_role and admin role

  admin_settings Table:
    ALL operations: role = 'admin' only
    service_role also has full access (for agent reads at runtime)

  agent_logs Table:
    SELECT: admin role only
    INSERT: service_role only

15.3  FASTAPI API SECURITY
  JWT Verification : All endpoints require Authorization: Bearer {token} header
                     FastAPI verifies against Supabase JWKS endpoint
  Role Extraction  : Decoded from JWT custom claim user_metadata.role
  Rate Limiting    : FastAPI middleware: 100 requests/minute per user (sliding window)
  CORS             : Allowed origins: {Vercel frontend domain only}
  HTTPS            : Enforced on both Railway (backend) and Vercel (frontend)
  Input Validation : Pydantic models on all FastAPI request bodies (auto-sanitized)

15.4  SERVICE ROLE KEY MANAGEMENT
  - Supabase service_role key stored ONLY as backend environment variable
  - Never exposed to Next.js frontend or any client-side code
  - Frontend uses anon key (public, safe to expose) — RLS restricts access
  - All mutations requiring service_role go through FastAPI backend endpoints

15.5  DATA PRIVACY
  Passwords        : Never stored by application (Supabase Auth handles bcrypt hashing)
  Payment data     : Not stored for COD (future card: delegate to PCI-compliant provider)
  GPS data         : rider_locations retained 30 days, then auto-purged
  Personal data    : Never logged in agent_logs (PII stripped before insert)

SECTION 18: DATA INGESTION & SCRAPING PIPELINE (APIFY)
================================================================================

18.1  RESTAURANT MENU SCRAPER

  Actor URL   : apify.com/its_raghav/restaurant-menu-scraper
  Input       : {startUrls: [{url: "..."}], maxItems: 500}
  Output/Item : {name, description, price, category, image_url, available, menuSection}
  Webhook     : On RUN SUCCEEDED → POST https://api.domain.com/ingest/apify/restaurant
  Payload     : {"body": {{outputBody}}}  (Apify template variable)

  FastAPI Handler Logic:
    1. Parse JSON array from webhook body
    2. Get or create store record for the restaurant
    3. Get or create category based on menuSection value
    4. For each item: UPSERT into products table
       (match on: name + store_id, update price/description/image on conflict)
    5. For new products: INSERT into inventory (quantity = 50 default for restaurant)
    6. Queue embedding generation: collect all new product_ids
    7. After upsert loop: call POST /ingest/generate-embeddings internally

18.2  GROCERY SCRAPER

  Actor URL   : apify.com/piotrv1001/instacart-scraper
  Input       : {storeCode: "...", category: "...", maxResults: 200}
  Output/Item : {name, description, price, unit, category, image_url, nutritional_info}
  Webhook     : On RUN SUCCEEDED → POST https://api.domain.com/ingest/apify/grocery
  Handler     : Same pattern as restaurant. nutritional_info stored as JSONB.
  Inventory   : Default quantity = 100 for grocery items.

18.3  EMBEDDING GENERATION PIPELINE

  Endpoint    : POST /ingest/generate-embeddings (internal, also called nightly by scheduler)
  Process:
    1. Query: SELECT id, name, description, tags, nutritional_info
               FROM products WHERE embedding IS NULL LIMIT 100
    2. For each product: build embedding text string:
         "{name}. {description}. Tags: {tags.join(', ')}. "
         + "Nutritional: calories={cals}, protein={prot}g, carbs={carbs}g."
    3. Batch call Gemini text-embedding-004:
         Send up to 100 texts per API call
         Model returns [{values: [0.123, -0.456, ...]}] per text (768 floats each)
    4. For each result: UPDATE products SET embedding = result.values WHERE id = product.id
    5. If more than 100 remaining: queue next batch (recursive background task)
    6. Log: total embedded, time taken, any failures

  Retry Logic: Products with NULL embedding after 3 attempts flagged in agent_logs.

SECTION 19: VECTOR SEARCH & SEMANTIC ENGINE (pgvector)
================================================================================

19.1  PGVECTOR EXTENSION SETUP
  Enabled in Supabase via: Extensions → pgvector (enable toggle in dashboard)
  SQL to run in Supabase SQL Editor:
    CREATE EXTENSION IF NOT EXISTS vector;
    ALTER TABLE products ADD COLUMN embedding vector(768);
    CREATE INDEX ON products USING ivfflat (embedding vector_cosine_ops)
      WITH (lists = 100);
    ANALYZE products;

19.2  SIMILARITY SEARCH QUERY TEMPLATE
  Search for similar products to a given query embedding ($1 = query vector):

    SELECT
      id, name, price, image_url, tags, store_id,
      1 - (embedding <=> $1::vector) AS similarity_score
    FROM products
    WHERE
      store_id = $2
      AND is_active = true
      AND (embedding <=> $1::vector) < 0.30  -- only results with >70% similarity
    ORDER BY embedding <=> $1::vector ASC
    LIMIT 10;

  Interpretation:
    cosine distance 0.0 = identical
    cosine distance 0.3 = 70% similar (good match threshold)
    cosine distance 1.0 = completely dissimilar

19.3  INDEX PARAMETERS
  Lists parameter: 100 (sqrt of estimated 10,000 products — good default)
  Probes at query time: SET ivfflat.probes = 10; (balance speed vs accuracy)
  For < 1,000 products: can use HNSW index instead for exact search with minimal overhead

19.4  EMBEDDING DIMENSION REASONING
  Gemini text-embedding-004 produces 768-dimension vectors.
  768 floats × 4 bytes = 3,072 bytes per product embedding.
  For 10,000 products: ~30MB for embedding column (fits Supabase 500MB free tier).
  IVFFlat index adds ~10MB overhead.

SECTION 25: PGVECTOR INDEX CONFIGURATION (SQL COMMANDS)
================================================================================

-- Enable extension (in Supabase SQL Editor or via supabase CLI migration)
CREATE EXTENSION IF NOT EXISTS vector;

-- Add embedding column to products table
ALTER TABLE products ADD COLUMN IF NOT EXISTS embedding vector(768);

-- Create IVFFlat index for fast approximate nearest neighbor search
-- (Run AFTER loading initial product embeddings — index builds on existing data)
CREATE INDEX products_embedding_idx
ON products
USING ivfflat (embedding vector_cosine_ops)
WITH (lists = 100);

-- Analyze table after indexing for query planner optimization
ANALYZE products;

-- Set probe count for query-time accuracy/speed trade-off
-- (Set in Supabase connection settings or per-query via FastAPI):
-- SET ivfflat.probes = 10;

-- Similarity search query (parameterized for FastAPI):
-- $1 = query embedding vector (string representation)
-- $2 = store_id UUID
-- $3 = similarity threshold (from admin_settings, default 0.25)
SELECT
  p.id, p.name, p.price, p.image_url, p.tags, p.description,
  1 - (p.embedding <=> $1::vector) AS similarity_score
FROM products p
INNER JOIN inventory i ON i.product_id = p.id
WHERE
  p.store_id = $2::uuid
  AND p.is_active = true
  AND i.quantity > 0
  AND (p.embedding <=> $1::vector) < $3
ORDER BY p.embedding <=> $1::vector ASC
LIMIT 10;

27.5  KEY TECHNICAL RISKS & MITIGATIONS

  RISK: Nominatim rate limit (1 req/second)
  MITIGATION: 300ms debounce on search input + 5-min client-side result cache
              + custom User-Agent header (required by Nominatim ToS)

  RISK: Supabase free tier DB size (500MB limit)
  MITIGATION: Compress embeddings (float16 instead of float32 saves 50%),
              nightly purge of rider_locations >30 days via APScheduler,
              archive orders >6 months to a compressed JSONB archive table

  RISK: GraphHopper 500 req/day limit
  MITIGATION: Cache route results keyed by (start_rounded_3dec, end_lat_lng),
              TTL 5 minutes. Most deliveries repeat similar origin-destination pairs.
              For prototype: 500 req/day = ~20-25 active delivery sessions per day.

  RISK: TensorFlow cold start on Railway (model load time ~10-15 seconds)
  MITIGATION: Pre-load model at FastAPI startup (lifespan event), keep in memory.
              Add /health endpoint that returns warm status. Railway keeps process
              alive between requests for free tier.

  RISK: LangChain agent latency (3-5 seconds for Gemini calls)
  MITIGATION: Show "AI is thinking..." loading state on frontend.
              Use Gemini Flash (fastest model) not Pro.
              Cache dispatcher scoring for same order+rider pairs within 5 minutes.

  RISK: Gemini API free quota (1M tokens/day)
  MITIGATION: Batch embedding generation (100 items per API call).
              Cache substitution reasoning per (original_product, candidate_product) pair.
              Monitor token usage in Google AI Studio dashboard.

  RISK: Race conditions during high-load checkout (multiple users buying last item)
  MITIGATION: PostgreSQL row-level lock on inventory (SELECT FOR UPDATE SKIP LOCKED)
              + soft reservation system with 15-minute hold window.
              Atomically decrement quantity inside a transaction block in FastAPI.

  RISK: Rider GPS accuracy on low-end phones
  MITIGATION: Use enableHighAccuracy: true in watchPosition options.
              Apply Kalman filter smoothing on received coordinates before storing.
              Dead-reckoning for brief GPS drops (extrapolate from last heading + speed).
